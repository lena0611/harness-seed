# 결과물 생성 흐름

이 문서는 요청을 받아 결과물이 나오기까지의 **고수준 판단 흐름**을 설명합니다.

> 주의: 내부의 상세 추론 전문을 그대로 기록하는 문서는 아닙니다.  
> 대신 어떤 단계로 판단하고 어떤 기준으로 다음 행동을 정하는지 이해하기 쉽게 정리합니다.

## 한눈에 보는 흐름

| 단계 | 질문 | 산출물 |
| --- | --- | --- |
| 1. 컨텍스트 복구 | 지금 저장소가 어떤 상태이고, 이전 세션에서 무엇이 남아 있는가? | 시작 기준선 |
| 2. 요청 해석 | 사용자가 실제로 원하는 결과는 무엇인가? | 작업 목표 |
| 3. 제약/규칙 로드 | 어떤 하네스, 개발 기준, 스타일, 문서 규칙이 먼저 적용되는가? | 적용 기준 |
| 4. 분류 | 이건 코드/문서/기준/운영/배포 중 무엇이 중심인가? | 작업 유형 |
| 5. 영향도 스캔 | 어떤 파일, 문서, 하네스, 워크플로가 함께 영향을 받는가? | 영향 범위 |
| 6. 강제 수준 판단 | harness, trigger, hook, block 중 어디까지 필요한가? | 적용 장치 |
| 7. 예외 범위 판단 | `none`, `defer`, `waiver` 중 무엇인가? | 예외 기준 |
| 8. 정보 부족 확인 | 지금 바로 판단 가능한가, 개발자 입력이 필요한가? | 질의 여부 |
| 9. 실행 계획 확정 | 무엇을 어떤 순서로 바꿀 것인가? | 작업 순서 |
| 10. 실행 | 문서 수정, 코드 수정, 스크립트/워크플로 추가 중 무엇을 할 것인가? | 실제 변경 |
| 11. 검증 | lint/build/check가 필요한가? | 검증 결과 |
| 12. 기록/인계 | 다음 세션이 이어받아야 할 내용이 있는가? | harness/queue/log 갱신 |

## 단계별 설명

### 1. 컨텍스트 복구
- 먼저 현재 세션이 어떤 저장소 상태에서 시작하는지 확인합니다.
- `session-harness`, `project-harness`, `policy-harness`, `documentation-harness`를 통해 기준선을 복구합니다.
- 이미 열린 미해결 항목, developer input queue, active context가 있으면 우선 반영합니다.

### 2. 요청 해석
- 요청의 표면 문장보다 **실제 원하는 결과**를 먼저 파악합니다.
- “문서를 추가해”처럼 보이더라도, 실제로는 운영 규칙/트리거/훅이 필요한지 함께 봅니다.

### 3. 제약/규칙 로드
- 현재 작업 전에 먼저 적용해야 할 구조 규칙, 스타일 규칙, 문서 규칙을 읽습니다.
- 새 규칙을 만들 때도 기존 하네스와 충돌하지 않는지 먼저 확인합니다.

### 4. 분류
- 코드 구현 중심인지
- 문서/기준 중심인지
- 세션 운영/개발 프로세스 중심인지
- 배포/검증 같은 실행 단계 중심인지

이 분류에 따라 읽을 하네스와 사용할 도구가 달라집니다.

### 5. 영향도 스캔
- 바꾸려는 대상 하나만 보지 않고, 같이 움직여야 하는 코드·문서·스크립트·워크플로를 찾습니다.
- 기준 변경이면 소스 영향도, 소스 변경이면 기준 영향도를 같이 봅니다.
- 문서가 길어지면 인덱스 분리 대상인지도 같이 판단합니다.

### 6. 강제 수준 판단
- 단순 방향 제시면 **Harness**
- 특정 상황에서 다시 떠올려야 하면 **Trigger**
- 자동 실행 또는 차단이 필요하면 **Hook**
- 차단이 필요한 경우는 **Block 수준**까지 고려

기준은 `.harness/policy/enforcement-ladder.md`를 따릅니다.

### 7. 예외 범위 판단
- 예외 없이 바로 지켜야 하는지
- 이번 세션만 유보 가능한지
- 제한된 waiver를 허용할지

이 역시 `enforcement-ladder.md`와 `waiver-guidelines.md` 기준으로 봅니다.

### 8. 정보 부족 확인
- 사용자 의도, 강제 강도, 예외 허용 범위가 분명하면 먼저 진행합니다.
- 아래 경우엔 사용자에게 묻습니다.
  - 강제 강도가 애매할 때
  - 예외 허용 범위가 애매할 때
  - 제품 목적/도메인 정보 없이는 위험할 때

정보 부족으로 멈춘 항목은 `developer-input-queue.md`로 넘깁니다.

### 9. 실행 계획 확정
- 바로 수정 가능한지, 먼저 구조를 세워야 하는지, 문서부터 바꿔야 하는지 순서를 정합니다.
- 한 파일 수정으로 끝나는지, 관련 레이어 전체를 함께 만져야 하는지도 여기서 결정합니다.

### 10. 실행
- 필요한 변경이 문서면 문서만 고치지 않고 인덱스/레지스트리도 같이 봅니다.
- 필요한 변경이 코드면 관련 policy guard, session harness 연결까지 봅니다.
- 필요한 변경이 운영 규칙이면 문서 + trigger + hook 중 어디까지 구현할지 같이 처리합니다.

### 11. 검증
- 문서만 바뀌어도 필요한 guard가 있으면 실행합니다.
- 현재 저장소에선 보통 `npm run harness:check`를 통합 검사 진입점으로 사용합니다.

### 12. 기록/인계
- 다음 세션이 이어받아야 하면 `active-context.md`, `decision-log.md`를 갱신합니다.
- 개발자 입력이 필요한 항목은 `developer-input-queue.md`에 남깁니다.
- 문서가 길어지면 documentation harness 기준으로 분리합니다.
- 사용자가 `세션종료`라고 말하면 `next-session-reminder.md`를 다음 세션용 요약으로 갱신합니다.
- `next-session-reminder.md`와 `active-context.md`에는 project 규칙 본문을 복사하지 말고, `workflow-rules.md` 등 권위 문서 포인터로 축약합니다.

## 이 저장소에서의 실제 적용 예

| 요청 유형 | 먼저 보는 것 | 보통 나오는 결과 |
| --- | --- | --- |
| 기준/규칙 추가 | `policy-harness`, `session-harness` | 문서 + trigger + 필요 시 workflow |
| 새 기능 구현 | `project-harness`, `policy-harness`, `session-harness` | core 우선 구현 + check 검증 |
| 문서 구조 정리 | `documentation-harness` | 인덱스 문서 + 하위 문서 분리 |
| 애매한 운영 규칙 | `enforcement-ladder`, `developer-input-queue` | 사용자 질의 또는 defer/waiver 기록 |

## 한 줄 요약

요청을 받으면  
**컨텍스트 복구 → 해석 → 규칙 로드 → 분류 → 영향도 스캔 → 강제 수준 판단 → 예외 판단 → 정보 부족 확인 → 계획 확정 → 실행 → 검증 → 기록/인계**  
순서로 움직인다고 보면 됩니다.
