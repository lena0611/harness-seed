# 개발자 입력 큐

개발자 정보 부족 때문에 완료되지 못한 항목을 새 세션마다 다시 확인하기 위한 큐입니다.

## 사용 목적
- 구현이나 구조 결정을 완료하려면 개발자 입력이 반드시 필요한 항목을 추적합니다.
- 새 세션 시작 시 이 큐를 보고 개발자에게 물어볼 항목을 우선 정합니다.
- 개발자가 답변을 미루고 싶을 때 그 상태를 명시적으로 기록합니다.

## 응답 선택 규칙
새 세션 시작 시 큐에 `open` 항목이 있으면 아래 셋 중 하나를 개발자에게 제안합니다.
1. **지금 답변**
2. **이번 세션에서는 유보**
3. **나중에 다시 묻기**

## 상태 정의
- `open`: 새 세션 시작 시 반드시 다시 확인해야 함
- `deferred`: 개발자가 이번 세션에서 답변을 유보함
- `answered`: 필요한 입력을 받아 해소됨
- `obsolete`: 더 이상 묻지 않아도 됨

## 현재 오픈 항목
| id | status | 질문 | 왜 필요한가 | 개발자 선택 |
| --- | --- | --- | --- | --- |
| charter-problem | open | 이 프로젝트가 해결하려는 핵심 문제는 무엇인가? | 프로젝트 헌장과 향후 feature 우선순위를 정하기 위해 필요 | 미정 |
| charter-users | open | 주요 사용자 또는 대상은 누구인가? | UI 흐름과 도메인 모델의 방향을 정하기 위해 필요 | 미정 |
| charter-success | open | 성공 기준은 무엇인가? | 완료 판단과 범위 통제를 위해 필요 | 미정 |
| charter-nongoals | open | 이번 프로젝트의 비목표는 무엇인가? | 범위 확산을 막기 위해 필요 | 미정 |

## 운영 원칙
- `open` 또는 `deferred` 항목은 `session-start-alert.md`에서 반드시 언급합니다.
- 개발자가 유보를 선택하면 해당 항목을 삭제하지 말고 `deferred`로 바꿉니다.
- 답변을 받으면 관련 문서(`project-charter.md`, `active-context.md`, `decision-log.md`)를 함께 갱신합니다.
- 강제 강도나 예외 허용 범위가 애매한 항목도 이 큐 또는 직접 질의 대상으로 올립니다.
