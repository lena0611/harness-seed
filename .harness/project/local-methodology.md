# 로컬 개발방법론

이 문서는 프로젝트 고유의 개발 방법을 모으는 진입점입니다.

공통 하네스는 공통 운영 기준을 제공합니다. 이 문서는 해당 프로젝트의 도메인, 아키텍처, 작업 흐름이 공통 기준 위에서 어떻게 해석되어야 하는지 기록합니다.

## 충돌 해석 우선순위

기준은 회사 공통, 스택, 템플릿, 프로젝트, 개인 순서로 쌓입니다. 하지만 실제 작업 중 충돌하면 아래 순서로 해석합니다.

1. 회사 공통 필수 차단 기준
2. 사용자의 명시 지시
3. 이 프로젝트의 로컬 개발방법론
4. 적용된 템플릿 사용 계약
5. 적용된 스택 기준
6. 회사 공통 기본 운영 기준
7. 개인 기준
8. 에이전트 기본값

회사 공통 필수 차단 기준은 보안, 권한, 검증, 복구, 기록, 차단처럼 피해를 제한하는 기준입니다. 로컬 방법론은 이 기준을 낮출 수 없습니다.

## 운영 원칙
- 로컬 방법론은 공통 하네스가 덮어쓸 대상이 아닙니다.
- 프로젝트 도메인 지식, 팀 관례, 릴리스 절차, 아키텍처 경계는 이 문서군에 남깁니다.
- 로컬 방법론이 회사 공통 기준, 스택 기준, 템플릿 사용 계약과 충돌하면 충돌 이유와 선택한 기준을 `decision-log.md`에 기록합니다.
- 개인 기준은 팀 공유 기준을 바꾸지 않습니다. 공유가 필요하면 프로젝트 기준으로 승격합니다.
- 아직 정해지지 않은 항목은 추측하지 않고 `developer-input-queue.md`에 질문으로 남깁니다.
