# 개발 기준 계층

공통 하네스는 기준을 한 덩어리로 덮어쓰지 않습니다. 회사 공통, 스택, 템플릿, 프로젝트, 개인 기준을 계층으로 보고 충돌을 드러낸 뒤 하나의 작업 맥락으로 정리합니다.

## 계층

| 계층 | 범위 | 예시 위치 | 커밋 여부 | 역할 |
| --- | --- | --- | --- | --- |
| 회사 공통 기준 | 모든 프로젝트 공통 | `.harness/`, `CLAUDE.md`, `AGENTS.md` | 커밋 | 스택과 무관한 개발 흐름, 검증 절차, 세션 운영 기준 |
| 스택 기준 | 특정 개발 스택 | `.harness/project/stack-preset-rules.md`, 외부 stack manifest | 커밋 | 공통 흐름을 특정 기술 스택의 구조와 명령으로 해석 |
| 템플릿 사용 계약 | 선택한 scaffold 템플릿 | `.harness/project/template-contract.md`, 템플릿 저장소의 개발 가이드 | 커밋 | 생성된 코드 묶음이 동작하기 위해 필요한 사용 계약을 프로젝트에 연결 |
| 프로젝트 기준 | 현재 저장소 팀 기준 | `.harness/project/*.md`, 기존 전용 하네스 문서 | 커밋 | 도메인, 아키텍처, 릴리스, 팀 관례 |
| 개인 기준 | 개인 작업 선호 | `CLAUDE.local.md`, `.claude/settings.local.json`, 개인 로컬 방법론 파일 | 커밋하지 않음 | 개인 도구 설정과 선호. 팀 기준을 바꾸지 않음 |

## 기준이 쌓이는 순서

1. 회사 공통 기준
2. 스택 기준
3. 템플릿 사용 계약
4. 프로젝트 기준
5. 개인 기준

이 순서는 설치와 이해의 기반 순서입니다. 회사 공통 기준은 모든 기준의 바닥이지만, 구체 작업에서 항상 더 구체적인 프로젝트 기준보다 우선한다는 뜻은 아닙니다.

## 충돌 해석 순서

1. 회사 공통 필수 차단 기준
2. 사용자의 명시 지시
3. 프로젝트 기준
4. 템플릿 사용 계약
5. 스택 기준
6. 회사 공통 기본 운영 기준
7. 개인 기준
8. 에이전트 기본값

회사 공통 기준은 두 층으로 해석합니다. 보안, 권한, 검증, 복구, 기록, 차단처럼 피해를 제한하는 기준은 `회사 공통 필수 차단 기준`으로 보고 사용자 지시나 프로젝트 기준보다 앞섭니다. 반면 읽기 순서, 문서 형식, 기본 운영 절차처럼 프로젝트마다 구체화될 수 있는 기준은 `회사 공통 기본 운영 기준`으로 보고 프로젝트/템플릿/스택 기준의 해석을 따릅니다.

프로젝트 기준이 템플릿이나 스택 기준보다 앞선다는 것은 적용 프로젝트가 최종 소유권을 가진다는 뜻입니다. 템플릿이나 스택 기준과 다르게 가려면 예외 범위와 이유를 `decision-log.md`나 `waivers.json`에 남깁니다.

개인 기준은 작업 편의를 돕는 보조 기준입니다. 프로젝트 기준, 스택 기준, 회사 공통 기준을 무시하는 근거가 될 수 없습니다. 생성 컨텍스트는 기준이 아니라 보조 산출물이므로 충돌 해석 우선순위에 넣지 않습니다.

## 충돌 처리

`npm run harness:scan`은 기준 출처를 모아 `.harness/session/project-scan-report.md`에 충돌 후보를 남깁니다.

충돌이 보이면 개발자는 다음 중 하나를 선택합니다.

1. 프로젝트 기준으로 승격해 팀 공통 맥락으로 고정한다.
2. 템플릿 계약이나 스택 기준을 따르되 프로젝트 예외를 문서화한다.
3. 개인 기준을 제거하거나 개인 로컬 파일에만 남긴다.
4. 임시 예외로 허용하되 범위와 만료 조건을 `waivers.json`에 남긴다.
5. 판단을 미루고 `developer-input-queue.md`에 질문으로 남긴다.

선택 결과는 `decision-log.md`에 기록합니다.

## 강제 장치

- 수동 개발자는 `npm run hooks:install`로 로컬 git hook을 선택 설치할 수 있습니다.
- 에이전트 작업은 hook 설치 여부와 무관하게 `CLAUDE.md`, 세션 boot, adapter hook을 통해 기준 확인과 `harness:check` 흐름을 따라야 합니다.
- 로컬 hook을 설치하면 `pre-commit`은 전체 `harness:check`, `pre-push`는 반복 검증을 줄이는 `harness:check -- --fast`를 실행합니다.
- CI나 보호 브랜치에서는 `npm run harness:check:strict`를 기준으로 차단합니다.
