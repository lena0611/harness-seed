# 프로젝트 하네스 작성 가이드

프로젝트 하네스는 현재 저장소의 팀이 공유해야 하는 개발 기준입니다. 공통 하네스와 스택 하네스가 제공하는 일반 기준 위에, 이 프로젝트에서만 의미 있는 도메인, 구조, 검증, 작업 절차를 쌓습니다.

## 역할

| 계층 | 담당 |
| --- | --- |
| 공통 하네스 | 모든 프로젝트가 공통으로 따를 AI 작업 흐름, 세션 복구, 문서 동기화, 검증 절차 |
| 스택 하네스 | 선택한 기술스택에서 반복되는 구조, 명령, 검증 기준 |
| 템플릿 사용 계약 | 선택한 scaffold 템플릿의 코드 묶음과 개발 가이드 사용 조건 |
| 프로젝트 하네스 | 현재 프로젝트의 도메인, 팀 관례, 아키텍처 결정, 릴리스 절차, 반복 검증 규칙 |
| 개인룰 | 개발자 개인의 도구 설정과 작업 선호 |

프로젝트 하네스는 공통/스택 기준을 덮어쓰는 별도 제품이 아닙니다. 적용된 기준을 현재 프로젝트의 실제 맥락으로 좁혀 주는 로컬 기준입니다.

## 언제 만드는가

프로젝트 하네스는 다음 상황에서 필요합니다.

- 기존 프로젝트에 회사 기준을 적용했지만 기존 팀 규칙과 도메인 지식이 흩어져 있을 때
- 스택 하네스만으로는 업무 도메인, 기존 구조, 릴리스 절차를 설명할 수 없을 때
- 버그 수정이나 기능 개발 중 같은 판단이 반복될 때
- 에이전트가 매번 코드를 다시 뒤져야 하는 프로젝트 고유 패턴이 있을 때
- 공통/스택/템플릿/개인 기준 사이에 충돌 후보가 있어 팀 기준으로 정리해야 할 때

반대로 한 번만 쓰는 임시 우회, 개인 선호, 아직 확인되지 않은 추측은 바로 프로젝트 하네스로 승격하지 않습니다.

## 작성 순서

1. 프로젝트에 맞는 스택 하네스를 먼저 설치합니다.
2. `npm run harness:scan`으로 현재 구조, 스타일 설정, 기존 문서, 충돌 후보를 확인합니다.
3. `project-charter.md`와 `scope-contract.md`에 프로젝트 목적과 범위를 채웁니다.
4. 기존 README, 위키, ADR, CI, 설정 파일, 반복 코드 패턴을 확인합니다.
5. 발견한 내용을 도메인, 아키텍처, 작업 흐름, 설정 계약으로 나눕니다.
6. 충돌이 있으면 `standards-layers.md`의 우선순위로 해석하고 `decision-log.md`에 선택 이유를 남깁니다.
7. 검증 가능한 규칙은 관련 테스트, lint, build, 문서 검사, 리뷰 체크리스트와 연결합니다.
8. `npm run harness:check`로 문서 링크와 기준 동기화 상태를 확인합니다.

## 파일별 책임

| 파일 | 적을 내용 |
| --- | --- |
| `project-charter.md` | 프로젝트 목적, 사용자, 성공 기준, 비목표 |
| `scope-contract.md` | 프로젝트가 다루는 범위와 명시적으로 다루지 않는 범위 |
| `local-methodology.md` | 프로젝트 하네스의 진입점과 충돌 해석 우선순위 |
| `standards-layers.md` | 공통/스택/템플릿/프로젝트/개인 기준의 계층과 충돌 처리 방식 |
| `stack-preset-rules.md` | 적용된 스택 하네스 기준. 관리 섹션은 직접 수정하지 않음 |
| `template-contract.md` | 적용된 scaffold 템플릿의 사용 계약 브리지. 전체 가이드는 템플릿 저장소가 소유 |
| `domain-rules.md` | 업무 용어, 핵심 엔티티, 도메인 불변식, 외부 시스템 계약 |
| `architecture-rules.md` | 모듈 경계, 의존 방향, 데이터 흐름, 공개 API 경계 |
| `workflow-rules.md` | 개발, 리뷰, 릴리스, 장애 대응, 검증 명령 |
| `commit-push-rules.md` | commit/push 승인 이후 실행되는 git hook, 커밋 템플릿, 최종 안전 검사 |
| `config-contract.md` | 환경변수, 설정 파일, secret, 런타임 설정 계약 |
| `decision-log.md` | 기준 충돌, 예외, 아키텍처 변경의 선택 이유 |
| `developer-input-queue.md` | 확인이 필요한 질문과 보류된 판단 |

프로젝트 하네스에서 가장 많이 쓰는 파일은 `domain-rules.md`, `architecture-rules.md`, `workflow-rules.md`입니다. 나머지는 기준을 찾고 충돌을 해석하기 위한 지지 문서입니다.

## 공통/스택 하네스와 어우러지는 원칙

- 공통 하네스의 세션, 문서, 검증 절차를 프로젝트 문서에서 다시 복사하지 않습니다.
- 스택 하네스의 관리 섹션을 직접 고치지 않습니다. 보충이나 예외는 관리 섹션 밖에 적습니다.
- 프로젝트 기준은 “우리 프로젝트에서는 왜 이렇게 하는가”를 설명합니다.
- 스택 기준과 다른 선택을 한다면 예외 범위, 이유, 검증 방법을 함께 적습니다.
- 회사 공통 안전 기준이나 사용자의 명시 지시를 프로젝트 기준으로 낮추지 않습니다.
- 개인룰은 팀 공유 기준보다 우선하지 않습니다. 반복 가치가 있으면 프로젝트 기준으로 승격합니다.
- 확신할 수 없는 내용은 규칙으로 고정하지 않고 `developer-input-queue.md`에 질문으로 남깁니다.

## 규칙 작성 형식

프로젝트 규칙은 짧아도 근거와 검증 방법을 같이 남깁니다.

```md
## <규칙 이름>

- 범위: 이 규칙이 적용되는 모듈, API, 배치, 작업 종류
- 근거: 기존 코드, 장애, 리뷰, 사용자 지시, 반복 패턴
- 기준: 앞으로 따라야 할 판단
- 검증: 확인할 명령, 테스트, 리뷰 항목, 문서
- 예외: 허용되는 예외와 기록 위치
```

이 형식을 강제 템플릿으로 쓰라는 뜻은 아닙니다. 다만 근거와 검증이 없으면 에이전트가 규칙을 맥락 없이 암기하게 되므로, 최소한 둘 중 하나는 남겨야 합니다.

## 예시

도메인 규칙:

```md
## 상태 변경 제한

- 범위: 상태 변경 API와 관련 작업 흐름
- 근거: 기존 운영 정책과 장애 대응 기록
- 기준: 종료 상태의 대상은 일반 변경 흐름에서 다시 활성화하지 않는다.
- 검증: 상태 변경 테스트와 예외 처리 경로 확인
- 예외: 운영 승인으로 복구해야 하면 별도 복구 절차와 decision-log에 사유 기록
```

아키텍처 규칙:

```md
## 외부 이벤트 재처리 안전성

- 범위: 외부 시스템 이벤트 수신과 동기화 작업
- 근거: 같은 외부 이벤트가 재전송될 수 있는 반복 패턴
- 기준: 외부 이벤트 처리에는 재처리 안전성을 확인할 수 있는 고유 키를 둔다.
- 검증: 중복 이벤트 처리 테스트와 저장 결과 확인
```

작업 흐름 규칙:

```md
## 외부 계약 필드 변경

- 범위: 외부 시스템이나 다른 저장소가 소비하는 데이터 계약
- 근거: 소비자가 같은 필드 계약을 공유해야 함
- 기준: 계약 필드를 추가, 제거, 이름 변경하면 contract 문서와 관련 테스트를 함께 갱신한다.
- 검증: 계약 테스트, 타입 검사, 문서 diff 확인
```

## 기존 프로젝트에서 만드는 방법

기존 프로젝트에서는 새 규칙을 처음부터 쓰기보다 이미 존재하는 기준을 흡수합니다.

1. `npm run harness:scan`의 `.harness/session/project-scan-report.md`를 확인합니다.
2. 기존 README, 위키, ADR, 에이전트 설정, CI 설정, 운영 문서를 기준 출처로 봅니다.
3. `.editorconfig`, ESLint, Prettier, tsconfig 같은 설정은 스타일 출처로 보고 프로젝트 규칙 초안에 반영합니다.
4. 코드에서 같은 패턴이 2곳 이상 반복되면 로컬룰 후보로 기록합니다.
5. 기존 전용 하네스 문서가 있으면 삭제하지 말고 `local-methodology.md`에서 연결합니다.
6. 충돌 후보는 즉시 하나로 덮지 말고 팀 선택이 필요한 항목으로 남깁니다.

## 승격 기준

작업 중 발견한 내용을 프로젝트 하네스로 승격할지는 다음 기준으로 판단합니다.

| 발견한 내용 | 처리 |
| --- | --- |
| 한 번뿐인 구현 세부사항 | 코드에만 둠 |
| 같은 판단이 반복되는 도메인 규칙 | `domain-rules.md` 후보 |
| 여러 모듈에서 반복되는 구조나 의존 방향 | `architecture-rules.md` 후보 |
| 리뷰마다 반복되는 확인 항목 | `workflow-rules.md` 후보 |
| 설정 파일에 이미 고정된 스타일 | 스타일 규칙 초안 또는 로컬룰 |
| 팀 합의가 필요한 충돌 | `developer-input-queue.md` 후 `decision-log.md` |
| 개인 작업 편의 | 개인룰에만 둠 |

승격은 “문서가 많아 보이게 하기 위해” 하는 일이 아닙니다. 다음 작업에서 같은 실수를 줄이거나, 에이전트가 추측하지 않아도 되게 만드는 정보만 올립니다.

## 검증과 유지보수

- 프로젝트 하네스를 바꾼 뒤에는 `npm run harness:check`를 실행합니다.
- 문서가 길어지면 README에 계속 붙이지 말고 인덱스와 하위 문서로 분리합니다.
- 룰 문서 상단에는 현재 따라야 할 요약과 적용 범위를 두고, 오래된 사례와 세부 근거는 하위 문서나 decision-log로 분리합니다.
- 에이전트가 모든 프로젝트 룰을 매번 읽는다고 가정하지 말고 `npm run harness:context -- "<작업 설명>"`으로 판단 기준과 충돌 우선순위를 좁히게 합니다.
- 규칙을 바꾸면 관련 테스트, CI, README, 운영 문서도 함께 봅니다.
- 더 이상 맞지 않는 규칙은 삭제보다 먼저 `decision-log.md`에 변경 이유를 남깁니다.
- 임시 예외는 범위와 만료 조건을 남기고, 가능하면 `waivers.json`에 기록합니다.

## 에이전트에게 맡길 때

에이전트에게 프로젝트 하네스 보강을 맡길 때는 다음처럼 지시합니다.

```text
현재 버그 수정을 하면서 발견한 반복 도메인/구조/검증 규칙이 있으면
.harness/project/*의 적절한 파일에 로컬룰 후보로 기록해라.
확신할 수 없는 내용은 규칙으로 확정하지 말고 developer-input-queue.md에 질문으로 남겨라.
```

이 지시가 있으면 에이전트는 공통/스택 기준을 먼저 읽고, 실제 코드와 작업 결과를 근거로 프로젝트 하네스를 조금씩 성장시킵니다.
