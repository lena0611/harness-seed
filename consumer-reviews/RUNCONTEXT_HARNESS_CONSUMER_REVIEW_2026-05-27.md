# RunContext 소비자 관점 하네스 총평

작성일: 2026-05-27

대상 프로젝트: RunContext / PaceLAB 러닝 코치 앱

목적: 하네스 본체 개발 관점에서, 실제 소비자 프로젝트에 하네스를 적용해 바이브코딩 방식으로 개발했을 때 충분한 가드레일이 되었는지 평가하고 개선점을 정리한다.

## 총평

이번 하네스는 없는 것보다 확실히 나았고, 장기 프로젝트에는 의미 있는 가드레일이 되었다.

다만 바이브코딩처럼 요청이 빠르게 누적되고 UI, 도메인, 배포, 네이티브, 백엔드가 계속 섞이는 상황에서는 아직 자동 개입력이 부족했다.

현재 수준은 검증 체크리스트, 프로젝트 기억 저장소, 커밋 전 안전장치로는 유효하다. 하지만 바이브코딩을 안정적인 제품 개발 프로세스로 자동 변환하는 시스템으로 보려면 더 개선이 필요하다.

## 좋았던 점

- `harness:check`가 테스트와 빌드를 묶어서 마지막 방어선 역할을 했다.
- 프로젝트 룰, 도메인 룰, HealthKit 데이터 계약, UI 시스템 계약 같은 문서가 쌓이면서 결정의 이유가 남기 시작했다.
- `critical-paths.md`, `manual-actions.md` 같은 장치가 생긴 뒤에는 Supabase, GitHub Pages, iOS capability처럼 에이전트가 직접 끝낼 수 없는 일을 구분하기 쉬워졌다.
- 하네스 업데이트 이후 `SYNC GAP`, 중요 경로, 정책 리뷰 메시지가 더 구체적으로 나와서 무엇을 확인해야 하는지 전보다 명확해졌다.
- 커밋 전 자동 검증, pre-push 훅, fast/full check 구분은 실사용에 현실적인 균형이었다.
- 소비자 프로젝트에서 하네스 업데이트 자체가 별도 커밋 단위로 정리되면서, 앱 기능 변경과 하네스 기준 변경을 분리해 볼 수 있었다.

## 부족했던 점

- 초반에는 에이전트가 하네스를 능동적으로 인식하지 못했다. 사용자가 하네스를 명시하기 전까지 룰, 메모리, 문서화 흐름이 자동으로 작동하지 않았다.
- UI 반복 버그, safe area, 상세 스택, 공통 컴포넌트 재사용 같은 패턴이 여러 번 반복된 뒤에야 룰화됐다. 사후 기록은 됐지만 사전 차단은 약했다.
- 하네스 메시지가 길고 노이즈가 많다. `harness:check` 출력은 정확하지만, 소비자 입장에서는 핵심 조치만 뽑아주지 않으면 읽기 어렵다.
- 바이브코딩 흐름에서는 지금 요청이 기존 룰과 충돌하는지, 공통 컴포넌트로 승격해야 하는지, E2E 추가가 필요한지를 에이전트가 매번 자동 판단해야 하는데 아직 강제력이 약했다.
- 디자인 시스템, 도메인 알고리즘, HealthKit 데이터 계약처럼 중요한 지식이 문서화되더라도 실제 코드 작성 순간에 항상 참조되지는 않았다.
- 반복되는 문제를 하네스가 감지해서 "이미 비슷한 문제가 있었다"라고 선제 경고하는 기능은 부족했다.

## 소비자 입장에서의 결론

하네스는 이번 프로젝트에서 검증 체크리스트, 프로젝트 기억 저장소, 커밋 전 안전장치로는 충분히 가치가 있었다.

그러나 선제적 가이드레일로는 아직 개선 여지가 크다.

소비자 입장에서 가장 원하는 것은 "내가 하네스를 의식하지 않아도 에이전트가 알아서 하네스를 참조하고, 반복 실수를 막고, 필요한 문서와 테스트를 자동으로 연결하는 것"이다.

## 개선 방향

### 1. 요청 시작 시 관련 문서 자동 축소

사용자 요청이 들어오면 에이전트가 전체 하네스 문서를 넓게 읽는 것이 아니라, 관련 문서 후보를 자동으로 좁혀야 한다.

예:

- UI 변경 요청: `ui-guidelines`, `ui-system-contract`, 공통 컴포넌트 규칙
- HealthKit 요청: `healthkit-data-contract`, 네이티브 수동 조치, 데이터 병합 정책
- Supabase 요청: config contract, manual actions, RLS/API key 정책
- 코칭 알고리즘 요청: running coaching standards, training knowledge ops

### 2. 반복 버그 자동 룰 후보화

같은 유형의 문제가 2회 이상 반복되면 하네스가 자동으로 로컬 룰 후보를 제안해야 한다.

예:

- iOS safe area 누락
- 바텀시트가 키보드나 home indicator에 가림
- 상세 스택에서 닫기/뒤로 정책 불일치
- 공통 리스트 컴포넌트 대신 화면별 중복 구현
- 숫자와 단위 typography 불일치

### 3. UI 변경 시 공통 컴포넌트 승격 체크

UI 요청이 들어오면 다음 질문을 에이전트 내부 체크리스트로 강제해야 한다.

- 기존 공통 컴포넌트로 가능한가?
- 새 컴포넌트가 필요하면 공통 컴포넌트로 승격할 가치가 있는가?
- 디자인 토큰을 사용했는가?
- 라이트/다크 테마를 모두 고려했는가?
- 모바일 safe area, 키보드, WebView 특성을 고려했는가?

### 4. E2E/단위 테스트 추가 여부 판단

기능 개발이나 버그 수정 후 하네스가 테스트 추가 필요성을 더 구체적으로 묻거나 판단해야 한다.

예:

- 라우트 이동 문제: E2E 필요
- 타입 추론 로직: 단위 테스트 필요
- HealthKit 병합 정책: 회귀 테스트 필요
- Supabase Edge Function: 함수 검증 또는 mock 테스트 필요

### 5. `harness:check` 결과 요약 개선

현재 출력은 상세하지만 길다. 소비자에게는 아래 형식의 요약이 필요하다.

```txt
결과: 통과
주의: SYNC GAP review suggested 3건
필수 조치: 없음
수동 조치: 없음
추천 조치: UI 룰 후보 1건 검토
검증: test/build 통과
```

상세 로그는 필요할 때 펼쳐보는 구조가 좋다.

### 6. 중요 경로 변경 시 검증 추천 강화

`critical-paths.md`는 좋은 방향이다. 다음 단계는 중요 경로별 추천 검증을 붙이는 것이다.

예:

- `supabase/functions/**`: Edge Function check, secret 노출 점검
- `src/shared/ui/**`: 라이트/다크, 모바일 viewport, 스냅샷 또는 E2E
- `src/features/infer-run-type/**`: 회귀 단위 테스트
- `ios/**`: Xcode 수동 빌드 필요, capability/manual action 확인

### 7. 에이전트가 하네스를 기본 루틴으로 인식

사용자가 "하네스"를 말하지 않아도 에이전트가 다음을 기본 루틴으로 해야 한다.

- 작업 전 관련 룰 확인
- 작업 중 반복 패턴 발견 시 룰 후보 기록
- 작업 후 테스트/빌드/harness check
- 커밋 전 diff와 수동 조치 여부 확인
- 최종 응답에 검증 결과와 남은 수동 조치 명시

## 하네스 본체 개발에 필요한 핵심 요구사항

1. 하네스는 사후 문서화 도구가 아니라 사전 개입 도구가 되어야 한다.
2. 에이전트가 하네스를 명시적으로 요청받지 않아도 자동 참조해야 한다.
3. 반복 문제를 감지하고 로컬 룰 후보로 승격해야 한다.
4. `harness:check`는 상세 로그와 별개로 소비자용 요약을 제공해야 한다.
5. 중요 경로별 추천 검증을 자동 제안해야 한다.
6. 수동 조치와 에이전트 조치의 경계를 더 선명하게 보여줘야 한다.
7. 바이브코딩 환경에서는 "빠른 요청 처리"와 "프로젝트 기준 준수"를 동시에 잡는 UX가 필요하다.

## 한 줄 결론

이번 하네스는 기록과 검증에는 유효했지만, 바이브코딩 중 반복 실수를 선제적으로 막는 가드레일로는 아직 부족하다. 다음 단계는 에이전트가 하네스를 자동으로 읽고, 반복 문제를 감지하고, 필요한 테스트와 문서를 선제적으로 연결하는 방향이어야 한다.
