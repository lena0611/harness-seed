# /업무요약

이번 운영 업무의 결과를 소비자 프로젝트의 유지보수 히스토리로 남깁니다.

일반적으로 사용자가 이 명령을 직접 외워서 실행하지 않아도 됩니다. `/운영업무`가 개발 완료 후보를 제시한 뒤 사용자가 완료 기록을 승인하면 이 절차를 이어서 수행합니다.

## 1. 기록 위치

연도별 폴더 아래에 `YYYY-MM-DD-<jira-key-or-summary-slug>.md` 형식으로 기록합니다.

예:

```text
.harness/maintenance/work-history/2026/2026-05-22-ABC-123.md
```

JIRA key를 URL에서 식별할 수 없으면 짧은 영문 slug를 사용합니다.

## 2. 기록할 내용

아래 형식을 사용합니다.

```markdown
# 업무 요약: <JIRA key 또는 짧은 제목>

- 날짜: YYYY-MM-DD
- 업무 유형: 버그픽스 | 신규 기능 | 단순 리팩토링 | 기존 기능 개선
- JIRA: <URL>

## 요청 개요
- <사용자 요청 또는 JIRA 요약>

## 핵심 개발 내용
- <변경한 핵심 코드/문서/설정>

## 검증
- <실행한 검증 명령과 결과>
- <실행하지 못한 검증과 이유>

## 잠재 이슈 또는 TODO
- <남은 위험, 후속 작업, 확인 필요 사항>
```

## 3. 함께 갱신할 수 있는 파일

히스토리를 쓴 뒤 아래 승격 여부를 반드시 판단합니다.

- 장기 운영 지식이면 `.harness/session/project-memory.md`
- 중요한 구조 결정, 예외 사유, 기준 충돌 해결이면 `.harness/session/decision-log.md`
- 다음 세션에서 이어야 할 일이면 `.harness/session/next-session-reminder.md`
- 반복 규칙으로 굳어졌으면 `.harness/project/domain-rules.md`, `architecture-rules.md`, `workflow-rules.md`

## 4. 기준

- 업무요약은 하네스 릴리스 노트가 아닙니다.
- 해당 소비자 프로젝트에서 어떤 운영 업무를 왜 어떻게 처리했는지 남기는 유지보수 히스토리입니다.
- 업무 히스토리는 팀 공유 대상이므로 Git 형상관리 대상입니다. 개인 메모나 비공개 정보는 넣지 않습니다.
- JIRA URL, 요청 개요, 핵심 개발 내용, 잠재 이슈 또는 TODO는 반드시 포함합니다.
